# Python项目结构规范

## 目录结构标准
project_root/

├── core/                 # 项目核心目录（可根据实际项目名称调整）
│   ├── api/              # API路由模块
│   ├── config/           # 核心配置和工具
│   ├── database/         # 数据库相关模块
│   ├── models/           # 数据模型
│   ├── services/         # 业务逻辑层
│   └── utils/            # 通用工具模块
├── tests/                # 测试文件目录
│   ├── api/              # API测试
│   ├── database/         # 数据库测试
│   └── services/         # 业务逻辑测试
├── config/               # 配置文件目录
├── .trae/                # Trae AI相关配置
├── requirements.txt       # 依赖管理
├── REFACTORING_SUMMARY.md # 重构总结（可选）
├── README.md             # 项目说明文档
└── main.py               # 应用入口文件（可选，根据框架选择）

## 依赖管理规范
- 使用 `requirements.txt` 明确声明所有依赖版本
- 所有依赖必须兼容Python 3.10+
- 为每个项目创建独立的虚拟环境

## 配置文件管理
- 配置文件放置于 `config` 文件夹中
- 在版本库中上传配置文件示例（如 `config.example.yaml`），不包含敏感信息
- 生产环境配置必须通过环境变量管理，禁止硬编码

## 环境变量管理
- 所有环境配置必须通过环境变量传递
- 环境变量命名采用大写字母加下划线分隔（如 `DATABASE_URL`）

## 测试文件结构
- 测试文件与源代码文件结构保持一致
- 测试文件命名采用 `test_*.py` 格式
- 使用 `pytest` 框架进行测试

## 文档规范
- 项目根目录下放置 `README.md`，包含项目概述、安装和使用说明
- 重要功能模块编写详细文档
- API文档根据所使用的Web框架选择合适的自动生成方式或手动编写

## 通用最佳实践
- 代码遵循团队约定的代码风格指南（参考coding_style.md）
- 为关键函数和方法添加类型注解，提高代码可读性和可维护性
- 保持代码结构清晰，便于扩展和维护
